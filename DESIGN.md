## Problem Statement

맥북 + 아이폰 유저가 사진을 아주 많이 찍는다. 사진 라이프사이클 전체가 고통이다:
- **백업**: iCloud가 금방 차서 외장하드로 옮겨야 하는데 번거롭다
- **골라내기**: 쓸데없는 사진, 중복, 비슷한 사진이 많아 삭제가 힘들다
- **분류**: 날짜별/이벤트별로 정리하고 싶은데 양이 너무 많아 엄두가 안 난다

핵심 고통: 사진 양이 늘수록 사람이 일일이 판단·정리해야 하는 노동이 선형적으로 늘어난다.

이번 세션에서 **첫 번째 주먹 = 날짜/이벤트별 자동 분류**로 좁혔다. 결과 형태는
**외장하드에 폴더 구조로 옮기고 끝**. 입력은 **이미 파일로 풀린 더미**(아이폰에서
내보냈거나 어떻게든 파일로 존재하는 상태).

## What Makes This Cool

흩어진 수만 장이 사람이 기억하는 단위로 착 정리되는 순간 — `2024-07_제주여행`,
`2024-09_동생결혼식` — 이게 "와" 포인트다. 핵심 마법은 **단순 날짜 폴더가 아니라
"이벤트 묶기"**다. 같은 7월이라도 "제주 여행 3일"과 "집에서 찍은 일상"을 다른
이벤트로 갈라준다. 거창한 AI 없이 시간 간격 + GPS만으로 잡힌다:
- 사진이 몰려 찍히다 몇 시간/며칠 뚝 끊기면 = 이벤트 경계
- 평소 집·회사 반경을 벗어난 좌표 = 여행/외출
- 역지오코딩 붙이면 폴더 이름(`제주`)이 자동으로 붙는다

## Constraints

- 플랫폼: macOS (맥북). 입력은 로컬/외장하드 파일.
- 입력 사진은 EXIF(촬영시간·GPS)가 살아있는 진짜 원본 파일이어야 한다.
- 사진은 두 번 다시 못 찍는 데이터 — 데이터 손실은 절대 불가.
- 사이드 프로젝트. "제일 빨리 멋지게 굴러가는 버전"이 목표. 주말 단위로 가치가 나와야 함.

## Premises

1. **진짜 차별점은 "날짜 폴더"가 아니라 "이벤트 묶기"다.** 단순 날짜 폴더는
   exiftool / osxphotos로 이미 해결됨. 시간갭+위치 기반 이벤트 분할에 집중해야 의미가 있음. — 동의
2. **입력은 best-effort EXIF — 없으면 신뢰하지 않고 격리.** 사진 앱 "Mac 저장공간
   최적화"가 켜져 있으면 원본이 디스크에 없음(썸네일만). → 이번 설계는 입력을 "이미
   파일로 풀린 더미"로 정해 이 함정을 우회. **단, 내보낸 더미는 파일 생성/수정시간이
   '내보낸 시각'이라 EXIF 대용으로 쓰면 안 됨**(전부 같은 가짜 이벤트로 뭉침). EXIF
   촬영시간이 없는 파일은 시간 추정 대신 `_분류안됨/`으로 격리. — 동의
3. **Live Photo / HEIC / 영상은 한 덩어리로 같이 움직여야 한다.** `IMG_1234.HEIC` +
   `IMG_1234.MOV` 짝을 인식해 함께 이동. — 동의
4. **첫 버전은 절대 move부터 하면 안 된다. dry-run + copy 우선.** "이렇게 옮길
   거예요" 계획만 보여주고 실제는 복사만, 검증 후 이동 옵션. — 동의

## Approaches Considered

### Approach A: exiftool 래퍼 + 시간갭 이벤트 분할 (미니멀, 이번 주말 완성)
Python ~150줄. exiftool로 촬영시간 읽고 → N시간 이상 끊기면 새 이벤트로 잘라서
→ `2024/2024-07_event01/` 폴더로 복사. UI 없음, dry-run으로 계획 출력 후 복사.
- 노력 S (사람 ~주말 / CC ~30분), 리스크 낮음
- 장점: 주말에 마법의 순간 확인 / exiftool이 메타·복사 검증 / 이벤트 로직 먼저 검증
- 단점: 이벤트 이름 밋밋(`event01`) / CLI 결과 검토 / 라이브포토 짝 직접 처리
- 재사용: exiftool, `-d` 날짜포맷

### Approach B: 이벤트 엔진 + 미리보기 UI (이상적, 진짜 제품)
시간갭 + GPS DBSCAN 클러스터링 → 오프라인 역지오코딩으로 지명 자동 →
라이브포토/HEIC 짝 인식 → 로컬 웹 미리보기에서 "이렇게 옮길게요" 보여주고
이름 수정·승인 후 복사/이동.
- 노력 L (사람 ~1~2주 / CC ~여러 세션), 리스크 중
- 장점: human-in-the-loop = 매번 쓰는 물건 / 중복제거·백업 얹는 토대 / 진짜 "와"
- 단점: 시간 대부분 UI에 / 스코프 커짐 / 한 번에 다 하려다 안 끝남
- 재사용: A의 엔진 그대로 내장

### Approach C: "사진 앱 두뇌 빌려쓰기" (창의적/측면)
입력을 파일 더미 대신 맥 사진 앱 라이브러리로 바꾸고, osxphotos로 애플이 ML로
만들어둔 이벤트·장소·인물 묶음을 추출해 그 경계대로 폴더 export.
- 노력 M (사람 ~3~4일 / CC ~1~2시간), 리스크 중
- 장점: 이벤트 품질 압도적 / 직접 짤 ML 거의 없음 / 인물·장소 메타 공짜
- 단점: 방금 정한 입력(파일 더미)을 뒤집어야 함 / 사진 앱 종속 / 최적화 함정 재등장

## Recommended Approach

**A로 시작하되 B를 목표로 한 단계적 계획** (사용자 선택).

이유: "제일 빨리 멋지게 굴러가는 버전"에 정확히 맞고, 차별점인 이벤트 묶기를 이번
주말에 눈으로 검증할 수 있다. A의 엔진이 그대로 B의 코어가 된다. C는 "이벤트 품질이
아쉽다" 싶을 때 꺼낼 카드로 보류.

- **마일스톤 1 (이번 주말, = Approach A):** 파일 더미 입력 → 시간갭 이벤트 분할
  → dry-run 계획 출력 → 복사. 라이브포토 짝 처리 + **파일명을 촬영시각 기반으로
  변경** 포함.
- **마일스톤 2 (다음, = Approach B 일부):** GPS 클러스터링 + 역지오코딩으로 이벤트
  이름 자동화(`event01` → `제주`).
- **마일스톤 3 (그 다음):** 로컬 웹 미리보기 + 이름 수정·승인 UI, 이동(move) 옵션.

## Open Questions

- 이벤트 경계 시간갭 임계값: **M1 기본값 4시간으로 고정**하고 실제 사진으로 사후
  튜닝. (자정 기준은 여행 새벽 사진을 끊어버리므로 안 씀.)
- **타임존**: `DateTimeOriginal`은 보통 타임존 없는 로컬시간. M1은 "찍힌 그대로의
  로컬시간"으로 정렬(여행 시차는 무시) — 이게 사람 기억과 대체로 맞음. 아이폰
  HEIC의 `OffsetTimeOriginal` 보정은 알려진 한계로 두고 나중에.
- 같은 파일이 이미 목적지에 있을 때(재실행) 중복 복사 방지 규칙: 1차는 파일명+크기
  일치 시 skip, 애매하면 해시.

(아래는 M1에서 결론을 내고 들어감 — 미정 아님)
- **EXIF 촬영시간 없는 파일**(스크린샷, 다운로드 이미지, 깨진 RAW): 파일시간으로
  추정하지 않고 `_분류안됨/`으로 격리. 시간을 모르니 **여기 파일은 이름을 바꾸지 않고
  원래 이름 유지**.
- **목적지 이름 충돌**(서로 다른 소스 폴더의 `IMG_0001.HEIC`): 충돌 시 원본 상대경로
  일부를 접미사로 붙여 보존, 덮어쓰기 금지.
- **파일명 변경 규칙**(신규): 기본 `YYYY-MM-DD_HHMMSS` (촬영시각). 같은 초 충돌은
  `_001`,`_002` 시퀀스. **라이브포토 짝은 동일한 새 basename 공유**(`...HEIC` +
  `...MOV`). 원래 `IMG_####` 번호는 기본적으로 버리되, `--keep-original-name`
  옵션으로 접미사 보존 가능(원본 추적용).

## Success Criteria

- **마일스톤 1 done:** 실제 수천 장 폴더를 입력하면, 사람이 "이 이벤트 묶음 맞네"라고
  고개 끄덕일 dry-run 계획이 나오고, --copy 실행 시 원본 손실 0으로 폴더 구조 생성.
- 결과 파일명이 `2024-07-15_143022` 식으로 정렬·식별 가능하게 바뀐다.
- 라이브포토 짝이 분리되지 않는다(새 이름도 짝끼리 일치).
- 같은 입력을 두 번 돌려도 중복 복사가 없다.

## Next Steps

1. **환경 준비:** `brew install exiftool`, Python 프로젝트 폴더 1개(`uv init` 권장).
   git repo로 만들어 두면 안전(현재 iCloud 폴더라 git 없음).
2. **읽기 단계 (성능 핵심):** 입력 폴더를 재귀 스캔 → exiftool을 **파일마다 부르지
   말고** `exiftool -json -r <폴더>` **한 번**으로 전체를 일괄 추출(5만 장이면 파일별
   호출은 몇 시간, 일괄은 수 분). 촬영시간 태그 우선순위는 파일종류별로:
   - 사진(HEIC/JPG): `DateTimeOriginal` → `CreateDate`
   - 영상(MOV/MP4): `CreationDate`(아이폰 로컬시간 포함) → `MediaCreateDate`/`CreateDate`(보통 UTC, 주의)
   - 위 태그가 전부 없으면 → **파일시간으로 추정하지 않고** `_분류안됨/` 격리 대상.
3. **짝 묶기:** 라이브포토 짝은 basename 대신 **`ContentIdentifier` EXIF 태그가 같은
   것**으로 묶는 게 견고(편집본 `IMG_E1234`, 버스트 때문에 basename 매칭은 깨짐).
   M1은 basename+같은 stem 매칭으로 시작하고, `ContentIdentifier`는 M1.5로.
4. **이벤트 분할:** 시간순 정렬 → 인접 사진 간 간격이 **4시간** 초과하면 경계. 각
   이벤트에 `YYYY-MM_eventNN` 이름 부여(월은 **그 이벤트 첫 사진의 월** 기준 — 연말
   여행이 두 달에 걸쳐도 한 폴더).
5. **dry-run 출력:** "이 사진 → 이 폴더" 매핑을 트리/CSV로 출력. `_분류안됨/`으로
   갈 파일 수도 함께 보고. **여기서 멈추고 눈으로 검증.**
5.5. **이름 만들기:** 각 파일(또는 라이브포토 짝)에 촬영시각 기반 새 이름 부여 —
   `YYYY-MM-DD_HHMMSS`, 같은 초 충돌은 `_NN` 시퀀스. **짝은 같은 basename 공유**.
   `_분류안됨/`행은 원래 이름 유지. exiftool의 `-d '%Y-%m-%d_%H%M%S%%-c'`로도 가능하나,
   짝 이름 일치를 보장하려면 자체 복사 로직에서 짝을 한 단위로 처리하는 게 안전.
6. **복사 실행:** `--copy` 플래그일 때만 실제 복사. 충돌 처리 순서를 명확히:
   목적지에 같은 이름 파일이 있으면 → **(a) 크기까지 같으면 동일 파일로 보고 skip**
   (재실행·중복 안전), **(b) 크기가 다르면 다른 파일이므로 접미사 붙여 둘 다 보존**
   (덮어쓰기 절대 금지). 같은 실행 안에서 서로 다른 소스의 동명 파일도 같은 규칙
   적용. 복사 후 크기 대조로 "원본 손실 0" 확인. 원본 절대 안 건드림.
7. (마일스톤 2) GPS → `reverse_geocoder`(오프라인) 또는 Nominatim으로 지명 자동.

## What I noticed about how you think

- "정리된 상태가 뭐냐"를 물었을 때 망설임 없이 **"외장하드에 폴더로 옮기고 끝"**을
  골랐다. 화려한 뷰어 앱이 아니라 끝나는 지점을 명확히 안다는 것 — 사이드 프로젝트를
  실제로 완성하는 사람의 사고방식이다.
- 세 고통 중 **분류**를 첫 주먹으로 골랐다. 가장 "아파 보이는" 저장공간 문제(백업)가
  아니라 매번 시간을 잘아먹는 진짜 병목을 짚었다.
- "A로 시작하되 B를 목표로"를 택했다. 다 만들고 싶은 욕심과 일단 굴리는 실용주의
  사이에서 후자를 골랐다 — 이게 끝나는 프로젝트와 안 끝나는 프로젝트를 가른다.

```
   │   소스총량 vs 목적지 여유 → 부족시 중단
   │   ★여기서 멈추고 눈으로 검증 (디스크 0바이트 쓰기)
   ▼
[7] --copy 실행 execute_copy()  [IO]
   │   .tmp에 쓰며 sha256 동시계산 → fsync → 되읽어 검증    ← D2,D3,N4
   │   검증 통과 → 원자적 rename → 최종경로
   │   충돌: 같은이름+같은크기 skip / 다른크기 둘다보존
   │   4GB초과(exFAT) → 격리                               ← D5
   ▼
[8] SQLite 맨페스트 기록  [IO]                              ← N3,D9
       원본경로·새경로·이벤트키·촬영시각·sha256·size·mtime
       ·exiftool버전·schema_version  (M2 다중축 인덱스 씨앗)
```
